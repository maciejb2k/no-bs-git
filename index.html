Boring preface

Git is hard for beginners. Do you have similar feelings? I think most of you can agree with that. Git was very hard for me, and after months of struggling with Git, I finally found out why.

In my opinion, the problem is that nobody starts learning Git with a full understanding of a .git/ directory.

Most of the tutorials and articles starts explaning Git with abstract concepts, like the commits are some "colorful dots", the branches are "dots connected together", like a metro stations on the map and so on.

I think these guides are absolute garbage, because when you need to think about solving a real problem with Git, your mind goes blank because you don't even know what it's all about.

Git is not a blackbox, which somehow works. I want this guide to provide sufficient basics, so you don't have to suffer every time something goes wrong in Git.

Before we start

You have to learn how to use Git from the command line. I don't care about your colorful GUI for Git with paid pink Barbie theme or some useless VSCode extensions. If you want to become a pro, you have to use it from the command line. No excuses.

Before going further, install Git and make sure it works by typing:
git --version

SHA1

Everything in Git is based on SHA1.

It is a cryptographic hash function. You don't have to know how it mathematically works. You have to remember, that it takes any piece of text and turns it into unique 40-character string of letters and numbers. It's impossible to get back the original text from this unique string.

If you are wondering, what if SHA1 will produce the same output for two different inputs, you can read about collisions of hash functions.

Spoiler: You will find that the probability of collision is extremly low. Also, you will learn that SHA1 is not secure anymore, but it doesn't matter, because it is not used for security in Git.
